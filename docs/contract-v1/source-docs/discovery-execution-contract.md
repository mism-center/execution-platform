# Discovery Portal / Execution Platform — Integration Contract

> Defines the interface between the MISM Discovery Gateway (producer) and the Execution Platform (consumer), mediated by the shared DAL (`mism-registry`).

---

## 1. Overview

The Discovery Portal and Execution Platform communicate through two mechanisms:

1. **Shared DAL (Postgres)** — both services connect to the same `mism-registry` database. The Discovery Portal writes `Resource` and `Run` records; the Execution Platform reads them.
2. **REST API call** — after preparing a Run in the DAL, the Discovery Portal sends the `run_id` to the Execution Platform's HTTP API to trigger execution.

**Discovery Portal** writes to the DAL: `register_model()`, `register_dataset()`, `prepare_run()`

**Execution Platform** reads from the DAL: `get_run()`, `get_resource()`, and updates run status (`start_run()`, `complete_run()`, `fail_run()`)

**REST trigger**: Discovery Portal → `POST /api/v1/runs` → Execution Platform

---

## 2. Pre-conditions: What the Discovery Portal Must Create

Before calling the Execution Platform, the Discovery Portal must ensure the following records exist in the shared DAL.

### 2.1 Model Resource

A `Resource` record with `resource_type = "model"` or `"tool"`.

| Field              | Required | Description                                                        |
|--------------------|----------|--------------------------------------------------------------------|
| `id`               | auto     | UUID, auto-generated by the DAL                                    |
| `name`             | yes      | Human-readable model name                                          |
| `resource_type`    | yes      | `"model"` or `"tool"`                                              |
| `location_uri`     | yes      | Location of the model artifact (e.g., iRODS path, URL)             |
| `execution_type`   | yes      | One of: `docker`, `conda`, `python`, `r`, `binary`, `huggingface`, `notebook`, `other` |
| `execution_ref`    | **yes**  | Container image reference (e.g., `containers.renci.org/mism/vivarium:latest`). The Execution Platform will reject models without this field. |
| `status`           | yes      | Must be `"active"` (the platform validates this)                   |
| `metadata`         | optional | May include `resource_requirements` (see Section 4)                |
| `io_spec`          | optional | If provided, the DAL validates input compatibility at `prepare_run` time |

Created via: `mism_registry.register_model()`

### 2.2 Input Dataset Resource(s)

Zero or more `Resource` records with `resource_type = "dataset"`.

| Field           | Required | Description                                                          |
|-----------------|----------|----------------------------------------------------------------------|
| `id`            | auto     | UUID, auto-generated by the DAL                                      |
| `name`          | yes      | Human-readable dataset name                                          |
| `resource_type` | yes      | `"dataset"`                                                          |
| `location_uri`  | **yes**  | iRODS path or storage URI. The Execution Platform mounts this into the container at `/input`. Rejected if missing. |
| `status`        | yes      | Must be `"active"`                                                   |

Created via: `mism_registry.register_dataset()`

### 2.3 Run Record

A `Run` record in `REGISTERED` status, linking the model to its inputs.

| Field                | Required | Description                                                     |
|----------------------|----------|-----------------------------------------------------------------|
| `id`                 | auto     | UUID, auto-generated by the DAL                                 |
| `model_id`           | **yes**  | References the model `Resource.id`                              |
| `input_resource_ids` | yes      | List of input dataset `Resource.id` values (may be empty)       |
| `status`             | auto     | Set to `"registered"` by `prepare_run()`                        |
| `model_version`      | auto     | Denormalized from the model's `version` field                   |
| `parameters`         | optional | Arbitrary key-value parameters for the run                      |
| `environment`        | optional | `RunEnvironment` (cpu, memory, gpu, env_vars, timeout)          |
| `triggered_by`       | optional | Identifier of who/what initiated the run (e.g., user ID, `"portal"`) |
| `notes`              | optional | Free-text; the Execution Platform writes the K8s `sid` here once launched |

Created via: `mism_registry.prepare_run()`

**Validation performed by `prepare_run()`:**
- Model exists and is `active`
- All input resources exist and are `active`
- If the model has an `io_spec`, input format compatibility is checked (IOSpec handshake)

## 3. Execution Platform API

### 3.1 Trigger Execution

```
POST /api/v1/runs
Content-Type: application/json
```

**Request body:**

```json
{
  "run_id": "<uuid from prepare_run>"
}
```

| Field    | Type   | Required | Description                        |
|----------|--------|----------|------------------------------------|
| `run_id` | string | **yes**  | ID of an existing Run in the DAL   |

**What the Execution Platform does on receipt:**

1. Fetches the `Run` from the DAL using `run_id`
2. Fetches the model `Resource` using `run.model_id` — extracts `execution_ref` (container image)
3. Fetches each input `Resource` using `run.input_resource_ids` — extracts `location_uri` values
4. Reads `resource_requirements` from `model.metadata` (falls back to defaults: 1 CPU, 2Gi memory)
5. Builds a Kubernetes pod spec with:
   - Container image from `execution_ref`
   - Input data mounted at `/input` (from iRODS PVC, using `location_uri` as sub-path)
   - Output directory mounted at `/output` (auto-generated path: `/mism/outputs/<run_id>`)
   - Environment variables: `MODEL_ID`, `RUN_ID`, `INPUT_PATH`, `OUTPUT_PATH`
6. Launches the pod on Kubernetes
7. Updates the Run status to `RUNNING` in the DAL

**Success response (201):**

```json
{
  "run_id": "abc-123",
  "sid": "k8s-session-id",
  "status": "running",
  "phase": null,
  "is_ready": null,
  "url": null,
  "error": null
}
```

**Error responses:**

| Status | Code                  | Cause                                                   |
|--------|-----------------------|---------------------------------------------------------|
| 400    | `validation_error`    | Run not found in DAL, model missing, no `execution_ref`, input resource missing or has no `location_uri` |
| 502    | `orchestration_error` | K8s pod launch failed                                   |

### 3.2 Query Run Status

```
GET /api/v1/runs/{run_id}
```

Returns the current status, enriched with live Kubernetes pod status when the run is active.

**Response (200):**

```json
{
  "run_id": "abc-123",
  "sid": "k8s-session-id",
  "status": "running",
  "phase": "running",
  "is_ready": true,
  "url": "https://...",
  "error": null
}
```

**Auto-status sync:** When queried, the platform checks the live K8s pod phase. If the pod has `succeeded` or `failed`, the DAL is automatically updated to `completed` or `failed`.

### 3.3 List All Runs

```
GET /api/v1/runs
```

**Response (200):**

```json
{
  "runs": [
    { "run_id": "abc-123", "sid": "...", "status": "running" },
    { "run_id": "def-456", "sid": "...", "status": "completed" }
  ]
}
```

### 3.4 Cancel a Run

```
DELETE /api/v1/runs/{run_id}
```

Cancels the run in the DAL and deletes associated K8s resources. Returns `204 No Content` on success.

## 4. Resource Requirements Convention

The model `Resource.metadata` dictionary may include a `resource_requirements` key to specify compute needs. If absent, defaults are used.

```json
{
  "resource_requirements": {
    "cpus": "2",
    "memory": "4Gi"
  }
}
```

| Key      | Format             | Default | Description                       |
|----------|--------------------|---------|-----------------------------------|
| `cpus`   | K8s quantity       | `"1"`   | CPU limit for the container       |
| `memory` | K8s quantity       | `"2Gi"` | Memory limit for the container    |

## 5. Run Status Lifecycle

- `registered` → `running` → `completed` (happy path)
- `registered` → `running` → `failed` (pod exits with error)
- `registered` → `cancelled` (cancelled before launch)
- `running` → `cancelled` (cancelled during execution)

| Transition                | Trigger                                     |
|---------------------------|---------------------------------------------|
| `registered` → `running`  | Execution Platform launches the pod          |
| `running` → `completed`   | Pod exits successfully (auto-detected on GET) |
| `running` → `failed`      | Pod exits with error (auto-detected on GET)   |
| `registered` → `cancelled`| DELETE /runs/{run_id}                        |
| `running` → `cancelled`   | DELETE /runs/{run_id}                        |

## 6. Container Contract

Model containers launched by the platform receive the following environment and mounts:

### Environment Variables

| Variable      | Value                            |
|---------------|----------------------------------|
| `MODEL_ID`    | The model Resource ID            |
| `RUN_ID`      | The Run ID                       |
| `INPUT_PATH`  | `/input`                         |
| `OUTPUT_PATH` | `/output`                        |

### Volume Mounts

| Mount Path | Source                                     | Access     |
|------------|--------------------------------------------|------------|
| `/input`   | iRODS PVC, sub-path from `location_uri`    | Read-only  |
| `/output`  | iRODS PVC, sub-path `/mism/outputs/<run_id>` | Read-write |

If multiple input resources are provided, they are mounted at `/input/0`, `/input/1`, etc.

---

## 7. Integration Sequence (End-to-End)

1. **Discovery Portal → DAL**: `register_model()`, `register_dataset()`, `prepare_run(model_id, input_ids)` → receives `run_id`
2. **Discovery Portal → Execution Platform**: `POST /api/v1/runs {"run_id": "..."}`
3. **Execution Platform → DAL**: `get_run(run_id)`, `get_resource(model_id)`, `get_resource(input_ids)`
4. **Execution Platform**: launches K8s pod
5. **Execution Platform → DAL**: `start_run(run_id)` — status becomes `running`
6. **Execution Platform → Discovery Portal**: responds `201 {run_id, sid, status}`
7. **Discovery Portal → Execution Platform**: polls `GET /api/v1/runs/{run_id}` for status updates
8. **Execution Platform → DAL** (on poll): if pod succeeded/failed, calls `complete_run()` or `fail_run()` and returns final status

---

## 8. Health Check

```
GET /healthz
```

Returns `{"status": "ok"}` when the Execution Platform is available. The Discovery Portal should verify this before sending run requests.
